Project Overview

Jazz Guitar Trainer は、Berklee College of Music の 7-Position System を完全に実装した ウェブベースの指板ビジュアライザーです。ギタリストがスケール、コードフォーム、ビバップフレーズを 体系的に学習できることを目的としています。

主要機能
  • 18モード対応 — Diatonic 7 + Melodic Minor 7 + Harmonic Minor 2 + Diminished 2
  • 7ポジション生成 — 12キー × 18モード = 216パターンの指板マッピング
  • コード進行モード — 1382曲のJazz Standards インポート、iReal Pro風譜面
  • ビバップフレーズ自動生成 — Charlie Parker スタイルの8音フレーズ
  • Drop 2/3 ボイシング — 20テンプレートによるコードフォーム表示
  • ガイドトーン表示 — 3rd/7th のボイスリーディング可視化
  • BPM自動再生 — メトロノーム、コード音声、フレーズ再生 (両モード共通)

Position Generation (7-Note Scales)

Berklee 7ポジションシステムの核心は B弦（2弦）が各ポジションで2音のみ という点。 他弦は全て3音。これはB-G弦間の長3度チューニング（他は完全4度）に起因します。

Step 1: Fret Map 生成 — buildFretMap()

6弦すべてについて、フレット0〜22のどこにスケール音があるかをマッピング。

開放弦チューニング (半音値): [4, 11, 7, 2, 9, 4]
                              1E   B   G   D   A   6E

各弦の各フレットについて (openString + fret) % 12 がスケール音に含まれるか判定し、 FretNote = [noteName, fret, semitone] のタプル配列として返します。

Step 2: Position 生成 — generatePositions()

不変ルール
このアルゴリズムは変更禁止。過去の失敗（Greedy matcher, ローテート, offset変更）を経て、 現在の「1:1順次割当」が唯一の正解として確立されています。
  1. B弦ペア取得: ルートペアをスキップし、2度ペアから7つの連続2音ペアを取得
  2. 他弦トリオ列挙: 各弦で最低フレットから3音の連続トリオを昇順に列挙
  3. 1:1 順次割当: trio[i] → pair[i] の単純マッピング
  4. STRING_DEG_OFFSETS を使って各弦の開始度数を決定: { e:3, g:5, d:2, a:6 }
  5. findClosestTrio(): フレット重複に基づきB弦ペアに最も近いトリオを選択 (maxGap=5)

C Ionian Pos 1 リファレンス

1E: F(1), G(3), A(5)     ← 3音
B:  D(3), E(5)           ← 2音 (常に)
G:  A(2), B(4), C(5)     ← 3音
D:  E(2), F(3), G(5)     ← 3音
A:  B(2), C(3), D(5)     ← 3音
6E: F(1), G(3), A(5)     ← 3音

Diminished Positions (8-Note Scales)

ディミニッシュ (W-H / H-W) は対称スケールのため 4ポジション × 2フェーズ = 8ポジション を生成。 dim7 の構成音 (Root, ♭3, ♭5, 6) は短3度間隔で等分布。

  • 各ポジションは5フレット幅、3フレット間隔
  • ポジション順: Pos 1=Root, Pos 2=♭3, Pos 3=♭5, Pos 4=6
  • 6E弦のみ4音、他弦は3音

Phrase Generator — Overview

フレーズジェネレーターは、リック検索型アーキテクチャでビバップフレーズを生成します。 WJD 186ソロ + Parker Omnibook 50ソロから抽出したリックライブラリから、 コード品質・拍数・ゴール音・スケール互換性に基づいて最適なリックを選択し、 指板座標上に解決(resolve)します。生成失敗時は null を返却します。

機能

  • リック検索型生成: リックライブラリからスコアリングで最適なリックを選択・指板解決
  • リック連結 (Lick Chaining): 1つ目のリック解決後、残り拍数 ≥1.0 なら2つ目のリックを自動連結。接合部は1-5半音以内に制限し、リック内部の94.5%と同等の自然さを維持
  • 可変リズム: 4分音符(q)、3連符(t)、8分音符(e)、16分音符(s)の混合に対応
  • 拍数選択: 通常モードで2/3/4拍のフレーズ長を選択可能
  • ゴールノート選択: 指板クリックでゴール音を指定し、フレーズの到達点を制御
  • ピアノロール: PhraseAnalysisPanel に時間×ピッチのSVG可視化を追加
  • リック境界表示: 連結フレーズで各ノートのリック帰属 (①/②/接続) を分析テーブルに表示。ピアノロールにも境界の垂直破線を描画
設計哲学
「リックのクオリティが高ければ、必然的に音楽的なビバップラインが出現する」— スコアリング自動生成を廃止し、リック検索に特化。リック連結 (Lick Chaining) により短いリックを自然に接続し、フレーズの充填率を向上。

Note Pool & Instance Scope

buildNotePool()

ポジション内の全スケール音を収集し、オプションでクロマチックアプローチ音を追加します。

PoolNote の構造
{
  noteName: string,       // "C", "D♭", etc.
  stringIdx: number,      // 0 (1E) ~ 5 (6E)
  fret: number,           // フレット番号
  semitone: number,       // 0-11 のピッチクラス
  isChordTone: boolean,   // コードトーン (R, 3rd, 5th, 7th)
  isApproach: boolean,    // クロマチックアプローチ音
}

Instance Scope Restriction

音域をコンパクトに保つため、第1インスタンスの範囲に制限します。

  • activePool: instFretMin - 1 〜 instFretMax + 1 のノートのみ
  • activeCtPool: activePool のうちコードトーンのみ
  • プール不足時 (≤6音 or CT<2) は全プールにフォールバック

Goal Note Selection

フレーズの最終音（beat 8）を最初に決定します。これが全体の方向性を決めます。

設計根拠
  • 70% の理由: パーカーの実際のソロでは ii-V-I フレーズ境界で 60-80% 程度の 7th→3rd 半音解決が観察される。100% だとフレーズが画一化するため 30% の変化余地を残す。
  • 半音（1 semitone）のみ: diff === 1 || diff === 11 で厳密に半音差のみチェック。全音は含まない。ii→V、V→I のような典型進行でのみ発動し、半音解決がない進行ではフォールバックが機能する。
  • 通常モードのゴール音: 次コードの 3rd そのものではなく、STRONG_RESOLUTION_DEGREE_IDX で推定した解決先の度数に最も近い現コードの CT（特に 3rd/7th）をゴールにする。ビバップでは現コード内で解決方向を示唆するのが定石。

chooseGoalNote() の決定ロジック

進行モード（nextChordContext あり）
  1. 現コードの7thが次コードの3rdに半音で解決する場合 → 70%の確率で現コードの7thをゴールに
  2. 次コードの3rdと1半音差のCTを優先検索
  3. フォールバック: 次コードの3rdと同名のCTを選択
通常モード

STRONG_RESOLUTION_DEGREE_IDX で「このモードが解決する先のコードの3rd」を推定。実際のゴール音は現コードのCTから選択:

ModeResolutionTarget
IonianI → IVIVの3rd (scale deg 6)
Dorianii → VVの3rd (scale deg 7)
MixolydianV → IIの3rd (scale deg 3)
Aeolianvi → iiiiの3rd (scale deg 4)

Lick Library — 抽出アルゴリズムと統計

フレーズジェネレーターの核となるリックライブラリは、2つのビバップソロデータベースから Python スクリプト (scripts/extract_licks_v2.py) で自動抽出されます。 計算音楽学の研究に基づく3段階のアルゴリズム (LBDM + MLA + DTL) で 音楽的に妥当なフレーズ境界を検出し、~11,000 リック を public/data/lick_library.json (~2.1 MB) として静的配信します。

参考研究とアルゴリズムの由来分類

本抽出パイプラインの各コンポーネントについて、先行研究との関係を分類しました。 「忠実度」は論文の手法をどの程度忠実に再現しているかを示します。

研究論文ベースのコンポーネント

手法論文実装箇所忠実度備考
LBDM Cambouropoulos, E. (2001). "The Local Boundary Detection Model (LBDM) and its Application in the Study of Expressive Timing" _lbdm_profile(), _lbdm_combined(), _pick_boundary_peaks() 高 論文の数式 (degree-of-change + proximity rule) をそのまま実装。重み (0.25/0.50/0.25) も論文デフォルト。閾値比率 0.5 とフレーズ最小音数 4 は独自チューニング

先行研究に基づくコンポーネント

以下のコンポーネントは当初「独自」と認識していましたが、調査の結果すべてに先行研究が存在します。

コンポーネント実装先行研究本実装の独自性
コード品質分類 classify_quality_omnibook(), classify_quality_wjd() コードシンボルの正規化は MIR の標準的手法。 Harte et al. (MIREX chord vocabulary) 等で体系化されている。 WJD 自体のアノテーションプロトコルにも品質分類基準が定義されている 5品質 (dom7/min7/maj7/min7b5/dim7) への縮約粒度はフロントエンド要件に基づく設計判断。正規表現ベースの閾値パーサーは独自実装
コード境界分割 split_at_chord_boundaries() Jazzomat プロジェクト の melpat ツールでフレーズをコーラス/フォーム部分単位で分割する手法が確立。 Frieler (2019) "Constructing Jazz Lines" でも「underlying chord の root を基準に pitch class を算出」=コード境界でコンテキストを切り替えることを前提としている LBDM フレーズをコードチェンジ位置でサブフレーズに分割する具体的な実装は独自だが、手法自体はジャズ分析研究の標準的アプローチ
リズム量子化 quantise_rhythm() Adams & Bartsch (2007) "Note segmentation and quantization for MIR" で MIR における量子化手法を体系化。 Cemgil et al. (2000) "Rhythm Quantization for Transcription" でベイズ統計に基づく確率的量子化モデルを提案。 DTL / Jazzomat では 1/6拍単位量子化 (duplet + triplet 対応) を使用 閾値ベースの離散化は MIR で標準的だが、4段階 (q/e/t/s) と具体的な閾値 (0.875, 0.417, 0.292) は独自チューニング。研究ではベイズ確率モデルがより高精度とされるが、リック抽出用途には十分
正規化 (リック形式変換) phrase_to_lick() Frieler (2019) が "chordal pitch classes" として (pitch − root) mod 12 (値 0–11) を定義。 DTL プロジェクト でもインターバル列をパターンの主要表現として採用 表現形式は Jazzomat / DTL の標準と同一。独自なのは出力 JSON フォーマット (フロントエンド互換) のみ
重複排除 deduplicate() Smith & Medina (2001) "Discovering Themes by Exact Pattern Matching" で完全一致パターンマッチング手法を研究。 Lartillot (2005) "Mining transposed motifs in music" で転調を考慮したパターンマイニングを提案。 Jazzomat melpat では closed patterns / cyclic patterns の概念で冗長性をフィルタリング (steps, rhythm) の完全一致キーによる重複排除は最も単純なアプローチ。DTL スコア + ソース優先度でのランキングは独自
構造バリデーション validate_phrase() Frieler MLA (2016) の統計: midlevel unit の中央値持続時間 2.25秒、lick の特性として「短くリズム的に多様」。 Jazzomat melpat では強拍開始/終了、フレーズ位置などの構造的制約でフィルタリング MLA の統計的知見と整合するが、フロントエンドの selectLick() 要件 (ノート数 3–20、拍数 0.5–8.5) に合わせた最小限の実装。Jazzomat はより多くの構造的制約を使用
独自性の所在

本パイプラインの各コンポーネントにはすべて MIR / Jazzomat / DTL の先行研究が存在し、 完全に独自のアルゴリズムはありません。本実装固有の設計判断は:

  • 5品質への縮約粒度 — フロントエンドのリック検索要件に由来
  • 各種閾値パラメータ — LBDM閾値比率、MLA分類閾値、リズム量子化閾値、DTL距離閾値はいずれも独自チューニング
  • パイプラインの組み合わせ — LBDM→コード境界分割→MLA→DTL という組み合わせ自体が独自の構成
  • フロントエンド互換の出力フォーマット — JSON 構造とフィールド名

抽出パイプライン全体像

1. ソロ読込 → 2. LBDM フレーズ境界検出 → 3. コード境界分割 → 4. リズム量子化 → 5. 構造バリデーション → 6. MLA 分類 → 7. 正規化 → 8. DTL スコアリング + 重複除去

Step 1 — ソロ読込

WJD (SQLite)

solo_info テーブルからスタイルが BEBOP / HARDBOP / COOL のソロを取得。 各ソロについて melody テーブル (pitch, onset, duration, bar, beat, beatdur) と beats テーブル (onset, chord) をクエリします。

Omnibook (MusicXML)

music21 ライブラリで MusicXML をパース。メロディパート (第1パート) から音符を、 ChordSymbol / Harmony 要素からコード進行を抽出します。 和音ノートは最高音のみ使用。テンポ情報 (MetronomeMark) から ビート長 (秒) を算出し、LBDM の IOI/レスト計算に使用します (デフォルト: BPM 160)。

Step 2 — LBDM フレーズ境界検出 (Cambouropoulos 2001)

LBDM (Local Boundary Detection Model) は、メロディの各遷移点について ピッチ・IOI (音符間時間) ・レスト (休符長) の3つのパラメータから 境界強度を計算し、局所的な極大値をフレーズ境界として検出します。

3つのプロファイル

プロファイル計算対象重み
Pitch隣接音間の |MIDI pitch差|0.25
IOI隣接音間の onset 差 (秒)0.50
Rest前の音の終了〜次の音の開始 (秒、0以上)0.25

IOI に最大重み (0.50) が割り当てられているのは、フレーズの切れ目は時間的な間隔で最も明確に現れるためです。

境界強度の計算 (Cambouropoulos 2001 の公式)

各プロファイルについて、遷移 i での境界強度 Si を次の2つのルールで計算します:

Change Rule (変化度)

隣接するインターバル間の変化比率:

r(k, k+1) = |x_k - x_{k+1}| / max(|x_k|, |x_{k+1}|)

直前のインターバルと比べてどれだけ変化したかを 0〜1 で表す。 例: [2半音, 7半音] の遷移 → r = |2-7|/max(2,7) = 5/7 ≈ 0.71 (大きな変化)。

Proximity Rule (近接度) × Change Rule
S_i = |x_i| × (r(i-1, i) + r(i, i+1))

インターバルの絶対値 (大きな跳躍ほど高い) に、 両側の変化度を掛け合わせる。つまり「大きなインターバルが、かつ前後と大きく異なる箇所」で 境界強度が最大になります。

重み付き合成

Combined_i = 0.25 × S_pitch_i + 0.50 × S_ioi_i + 0.25 × S_rest_i

境界選択: 局所極大値のピック

合成された境界強度プロファイルから、局所極大値 (両隣より大きい点) を検出し、 最大強度の 50% 以上の点のみをフレーズ境界として採用します。 各境界でソロを分割し、4音以上のフレーズのみを保持します。

なぜ LBDM か

v1 ではスライディングウィンドウ方式 (3〜16音の全部分列を列挙) を使用していましたが、 41万個の候補からアドホックなフィルタ (音域/反復/跳躍) で除去する必要がありました。 LBDM はピッチ・時間・休符の情報から音楽的に妥当な境界を直接検出するため、 アドホックフィルタが不要になり、抽出されるリックの品質も向上します。

Step 3 — コード境界分割

LBDM で検出されたフレーズを、さらにコード変更点で分割します。 各ノートの onset を見て、そのタイミングで有効なコードを特定し、 コード変更が起きた箇所でサブフレーズに分割します。

LBDM フレーズ: [───────── phrase ─────────────────] コードタイムライン: Dm7 (onset=0) ────── G7 (onset=4) ────── └── min7 サブフレーズ ─┘ └── dom7 サブフレーズ ─┘

これにより、各リックは特定のコード品質のコンテキスト内に収まり、 品質別に分類されたライブラリが構築されます。

Step 4 — リズム量子化

音符の duration を4種類のリズムタイプに量子化します。 WJD では duration / beatdur (ビート相対値)、Omnibook では quarterLength を使用。

Step 5 — 構造バリデーション

LBDM がフレーズ境界を正確に検出するため、音楽的品質フィルタ (音域/反復/跳躍) は不要です。 フロントエンドの selectLick() が要求する最小限の構造制約のみチェックします:

制約条件リジェクト数
ノート数3 ≤ length ≤ 20WJD: 3,037 / Omnibook: 444
拍数0.5 ≤ durationBeats ≤ 8.5WJD: 98 / Omnibook: 47

Step 6 — MLA 分類 (Frieler/Pfleiderer 2016)

Melodic Lick Analysis (MLA) の基準でフレーズを lick / line / other に分類し、 line (リズム的に均一なスケーラーラン) を除外します。

分類条件ライブラリへの採否
line 16音超 かつ 最頻リズムが全体の 70%以上 除外 (WJD: 87 / Omnibook: 16)
lick 16音以下、またはリズム種類が2種以上 採用
other 上記に該当しない 除外
lick vs line の区別

ビバップソロには、コード上を駆け上がる/駆け下りるスケーラーラン (line) が頻出します。 これらはリズム的に均一 (ほぼ全て8分音符など) で長いパッセージであり、 短いフレーズ生成に再利用する「リック」としては不適切です。 MLA の基準で除外することで、リズム的多様性のあるリックのみをライブラリに残します。

Step 7 — 正規化

バリデーションと MLA を通過したフレーズを、キーに依存しない形式に正規化します。

リックのデータ構造
{
  id:            string,    // MD5ハッシュ先頭12文字 (重複除去キー)
  steps:         number[],  // ルートからの半音オフセット (0-11)
  intervals:     number[],  // 隣接音間のインターバル (符号付き半音)
  rhythm:        string[],  // リズムタイプ配列 ['e','e','s','s','e']
  direction:     string,    // 'asc' | 'desc' | 'mixed'
  length:        number,    // ノート数
  startStep:     number,    // 開始音の steps[0]
  endStep:       number,    // 終了音の steps[last]
  durationBeats: number,    // 総拍数 (リズムから計算)
  source:        string     // 'wjd' | 'omnibook'
}

steps (ルート相対半音) の計算

各ノートの MIDI ピッチからコードのルート PC (0=C, 2=D, ..., 11=B) を引き、 mod 12 で 0〜11 に正規化します。これにより:

  • 異なるキーで演奏された同一パターンが同じ steps 配列になる
  • フレーズジェネレーターが任意のキーに移調して再利用できる
  • steps[i] = 0 はルート、4 は長3度、7 は完全5度、など

direction の判定

値条件
asc全インターバルが ≥ 0 (上行または同音)
desc全インターバルが ≤ 0 (下行または同音)
mixed上行と下行が混在

Step 8 — DTL パターン頻度スコアリング + 重複除去

DTL パターン頻度分析

各品質カテゴリ内で、リックのインターバル列を Levenshtein距離 でクラスタリングし、 複数ソロに出現するパターンにボーナスを付与します。

  1. 各リックのインターバル列 (intervals) をタプル化
  2. 同じ長さ ± 1 のインターバル列をバケット化し、バケット内で Levenshtein距離を計算
  3. 距離 ≤ 2 のペアを Union-Find でクラスタリング
  4. クラスタサイズ ≥ 3 → 最大 +10 ボーナス
  5. 複数ソースにまたがるクラスタ → +15 ボーナス (WJD と Omnibook 両方に出現)
DTL の意義

複数の異なるソロに共通して出現するパターンは、ビバップの「定型句」である可能性が高い。 Levenshtein距離を使うことで、完全一致だけでなく1〜2音の変形を含む類似パターンも 同一クラスタとして扱い、イディオムの検出精度を高めます。

重複除去

(steps, rhythm) のタプルが同一のリックを重複とみなします。 重複がある場合は DTL スコア + ソース優先度 (Omnibook +5) が高い方を保持します。

品質ごとの上限 (max_per_quality = 8,000)

重複除去後のリック数が 8,000 を超える品質カテゴリでは、 DTL スコア順に上位を選択します。現在どの品質も上限に達していません。

コード品質分類

各コードシンボルを 5つの品質カテゴリ に分類します。

品質キー対象コード例分類ルール
dom7C7, G7sus, F9dominant-seventh、接尾辞なし、sus、+7 等 (デフォルト)
min7Dm7, Am, Cm9接尾辞に - / m / minor を含む
maj7Cmaj7, F6, G69接尾辞に maj7 / j7 / 6 / 69 を含む
min7b5Bm7b5, Aø7m7b5 / ø (half-diminished)
dim7Bo7, Cdim7, Edimdim / o を含む

WJD と Omnibook で別々の分類関数を使用しますが、出力する5品質キーは共通です。 スラッシュコード (例: Dm7/G) はスラッシュ前のコード本体 (Dm7) で品質・ルートを判定し、分数ベース音 (G) は無視します。 判別不能なシンボルは dom7 にフォールバック（ジャズの慣例に従う）。

Lick Selection & Resolution

フレーズ生成の中核。リックライブラリ (~11,000リック) から条件に最適なリックを検索し、 指板上の物理的な座標に解決します。スコアリングによる自動生成は行わず、 実際のビバップソロから抽出したリックの品質に依存するアーキテクチャです。

設計根拠
リックの品質が高ければ、スコアリングで1音ずつ生成するより遥かに自然なフレーズが得られる。 WJD 186ソロ + Omnibook 50ソロから LBDM + MLA + DTL で抽出した ~11,000 リックは、ビバップの語彙を十分にカバーしている。 失敗時は null を返却し、UI は前回のフレーズを維持する。

selectLick() — リック選択

品質・拍数・スケール互換性に基づいてリックを選択します。

  1. 品質フィルタ: toLickQuality() でコード品質からリック品質キーに変換
  2. durationBeats 制約: 指定拍数に一致するリックのみ
  3. スケール互換フィルタ: modeSemi (現在のスケールの半音セット) と照合。uniqueOut ≥ 2 のリックは除外、outCount × -15 のペナルティ
  4. チェインフィルタ (2つ目のリック選択時): chainFromStep が指定されている場合、startStep との pitch-class 距離が 1-5半音以内のリックのみ通過。同音反復(0)と跳躍(>5)を除外。リック内部インターバルの94.5%が≤5半音であることに基づく閾値
  5. 特性音ボーナス: CHARACTERISTIC_TONES に含まれる音を持つリックに +20
  6. 追加スコアリング: ゴール互換 / startHint 近接 / コンター親和 / 充填率
  7. Top-10 重み付きランダム選択

resolveLick() — 指板解決

選択されたリックの各ステップを、ノートプール内の物理的な指板位置に解決します。

  • 各ステップで targetSemitone に一致する候補をスコアベースで選択
  • スコア要素: absolutePitch 近接 + 弦距離 + 方向一致
  • 品質ゲート: 強拍アプローチ音 → reject、連続音間 >9半音 → reject
  • 最大5リトライ、全失敗 → null 返却

コネクタ音

リック解決後、ゴール音に到達していない場合はコネクタ音を追加。 getApproachNotes() で半音アプローチ / エンクロージャーを生成します。 アプローチタイプのチェックボックスは将来のコネクタ機能拡張用に UI を保持していますが、現在は無効化されています。

Chord Forms (Drop 2 / Drop 3 Voicings)

各ポジション内で弾ける Drop 2 / Drop 3 ボイシングを自動検出し、指板上にハイライト表示。

ボイシング理論

  • Drop 2: 4-way close voicing の上から2番目の音を1オクターブ下げる → 4連続弦
  • Drop 3: 上から3番目を1オクターブ下げる → 1弦スキップの4弦
  • 転回形はベース音で命名（ジャズ標準）: Root, 1st (3rdベース), 2nd (5thベース), 3rd (7thベース)

Audio Engine

すべて Web Audio API ベース。外部ライブラリ依存なし。

  • Karplus-Strong (Guitar) — 物理モデリングによるギター単音シンセ (playKSNote()): ノイズ励起 → 6-pass移動平均 → KSループ (0.4/0.6 LP + decay 0.996) → LP 2200Hz + HighShelf -6dB
  • Sawtooth + Formant (Saxophone) — リード楽器シンセ (playSaxNote()): Sawtooth発振 → 並列バンドパス (F1=500Hz body, F2=1400Hz reed) → LP warmth → Sine LFO 5.5Hz ビブラート (150ms遅延)
  • Electric Piano (Chord) — エレピシンセ (playEPNote()): Sine基音 + 2nd倍音 (ウォーム減衰) + 3rd倍音 (アタックのみ) → LP 4500Hz。コードストラムに使用
  • Instrument Dispatcher — playNote(ctx, freq, vol, time, dur, instrument) でフレーズ楽器別にディスパッチ (guitar/saxophone)
  • Chord Strum — ボイシングの同時/アルペジオ発音 (playChordStrum()) — ローズピアノ音
  • Click — メトロノーム用オシレーター (playClick())
  • Phrase Scheduling — schedulePhrase() で Web Audio API タイムスタンプに基づく正確な発音 (instrument パラメータ対応)

Data Flow

JazzStandards.json ↓ fetchJazzStandards() RawJazzStandard { Sections: RawSection[] } ↓ extractStructuredChords() StructuredSection { measures: MeasureChord[][], endings?, repeats? } ↓ songToProgression() Progression { chords: ChordSlot[], chartLayout: ChartLayout } ↓ ChordChart component CSS Grid 描画

Design Decisions

  • chords: ChordSlot[] フラット配列 = ロジックのsingle source of truth
  • chartLayout?: ChartLayout = 表示用メタデータ（chords[] へのインデックス参照）
  • ユーザー作成進行（layoutなし）→ deriveChartLayout() で1コード/小節に自動生成
  • beatWidths: フレックス比率 — ChordChart で flex: beats, BPM 再生で (bw/bwSum)*4

Fretboard SVG

SVG ベースの指板描画。6弦 × 22フレットのグリッド上にノート、ゴーストノート、ボイシング、 フレーズパスを重ね描画します。

Phrase Path Rendering

PhrasePath.tsx — Catmull-Rom スプラインを Cubic Bezier に変換して SVG <path> で描画。

  • グラデーション: per-beat (#FFA0B0 → #BBA0FF)
  • ノート形状:
    • CT = 塗り潰し円
    • アプローチ = ダイヤモンド
    • スケールトーン = 枠円（ストロークのみ）
  • 拍番号: 各ノートの横に表示
  • アニメーション: 各ノートのフェードインは beatStart ベースのタイミング（リズムに忠実な表示）
  • 視認性改善:
    • Crowding-aware routing: 各セグメントの中点近接度を距離減衰で計測し、密集エリアの線を垂直方向にファンアウト（最大3層）
    • タンジェント抑制: 短セグメント（同フレット弦移動等）は Catmull-Rom タンジェントをセグメント長比例で減衰させ、ほぼ直線に
    • 折り返しbow抑制: 近接フレット間の方向反転でのbow量をセグメント長に比例してスケール
    • 再訪問マーカー: 75%縮小 + 次ノート方向に6pxオフセット
    • ストローク幅テーパリング: 始点2.2px → 終点1.4pxで時間方向を明示
    • 開始ノート強調: マーカー1.2倍 + 半透明リング

Type Definitions

全型は src/types/music.ts に定義。主要な型:

Core Types (展開) 
// 指板上の音
type FretNote = [noteName: string, fret: number, semitone: number]

// ポジション
interface Position {
  id: number
  bPair: FretNote[]        // B弦の2音
  range: [min: number, max: number]
  instances: PositionInstance[]
}

// モード
interface Mode {
  key: string              // "ionian", "dorian", etc.
  name: string             // Display name
  semi: number[]           // Semitone pattern [0,2,4,5,7,9,11]
  notes: string[]          // Note names ["C","D","E",...]
  degrees: string[]        // Degree labels
  chord: string            // Chord symbol
  chordQuality: string     // "maj7", "m7", "7", etc.
  chordTones: string[]     // [R, 3rd, 5th, 7th]
  chordSub?: string        // Subscript suffix
}

// フレーズノート
interface PhraseNote {
  noteName: string
  stringIdx: number        // 0-5
  fret: number
  semitone: number         // 0-11
  isChordTone: boolean
  isApproach: boolean
  isExtension?: boolean
  beatPosition: number     // 1-8
  isStrong: boolean        // beats 1,3,5,8
  approachGroup?: ApproachGroupInfo
  digitalPattern?: DigitalPatternTag   // デジタルパターンのタグ
  isBebopPassing?: boolean             // ビバップ経過音
  isSkeletonBeat?: boolean             // スケルトン拍
  duration?: RhythmType                // 音価 (q/t/e/s)
  beatStart?: number                   // 絶対拍位置 (0起点)
  lickIdx?: number                     // 0=1st lick, 1=2nd lick (連結時), undefined=コネクタ
}

type RhythmType = 'q' | 't' | 'e' | 's'  // 4分/3連/8分/16分
type InstrumentType = 'guitar' | 'saxophone'  // フレーズ再生楽器

// リック型
interface Lick {
  id: string
  steps: number[]          // interval-from-root (0-11)
  intervals: number[]      // signed semitone intervals
  rhythm: RhythmType[]     // duration per note
  direction: 'asc' | 'desc' | 'mixed'
  length: number
  durationBeats: number
  source: 'omnibook' | 'wjd'
}

// 生成メタデータ型
interface SkeletonMeta {
  patternLabel: string     // "R→3→5→7"
  direction: 'asc' | 'desc' | 'mixed'
  continuityCtIdx?: number // startHintから決定されたbeat1 CT index
}
interface DigitalPatternTag {
  name: string             // "1-2-3-5"
  position: number         // 0-based index
  size: number             // pattern length
}

// コード進行
interface Progression {
  name: string
  songKey?: SongKey
  chords: ChordSlot[]
  chartLayout?: ChartLayout
}
