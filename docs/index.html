Project Overview

Jazz Guitar Trainer は、Berklee College of Music の 7-Position System を完全に実装した ウェブベースの指板ビジュアライザーです。ギタリストがスケール、コードフォーム、ビバップフレーズを 体系的に学習できることを目的としています。

主要機能
  • 18モード対応 — Diatonic 7 + Melodic Minor 7 + Harmonic Minor 2 + Diminished 2
  • 7ポジション生成 — 12キー × 18モード = 216パターンの指板マッピング
  • 練習モード — 1382曲のJazz Standards インポート、iReal Pro風譜面
  • ビバップフレーズ自動生成 — ルールベースエンジンによるビバップフレーズ生成
  • Drop 2/3 ボイシング — 20テンプレートによるコードフォーム表示
  • ガイドトーン表示 — 3rd/7th のボイスリーディング可視化
  • BPM自動再生 — メトロノーム、コード音声、フレーズ再生 (両モード共通)
  • スウィングモード — タイミング+ダイナミクス+アーティキュレーションの多次元スウィング

Position Generation (7-Note Scales)

Berklee 7ポジションシステムの核心は B弦（2弦）が各ポジションで2音のみ という点。 他弦は全て3音。これはB-G弦間の長3度チューニング（他は完全4度）に起因します。

Step 1: Fret Map 生成 — buildFretMap()

6弦すべてについて、フレット0〜22のどこにスケール音があるかをマッピング。

開放弦チューニング (半音値): [4, 11, 7, 2, 9, 4]
                              1E   B   G   D   A   6E

各弦の各フレットについて (openString + fret) % 12 がスケール音に含まれるか判定し、 FretNote = [noteName, fret, semitone] のタプル配列として返します。

Step 2: Position 生成 — generatePositions()

不変ルール
このアルゴリズムは変更禁止。過去の失敗（Greedy matcher, ローテート, offset変更）を経て、 現在の「1:1順次割当」が唯一の正解として確立されています。
  1. B弦ペア取得: ルートペアをスキップし、2度ペアから7つの連続2音ペアを取得
  2. 他弦トリオ列挙: 各弦で最低フレットから3音の連続トリオを昇順に列挙
  3. 1:1 順次割当: trio[i] → pair[i] の単純マッピング
  4. STRING_DEG_OFFSETS を使って各弦の開始度数を決定: { e:3, g:5, d:2, a:6 }
  5. findClosestTrio(): フレット重複に基づきB弦ペアに最も近いトリオを選択 (maxGap=5)

C Ionian Pos 1 リファレンス

1E: F(1), G(3), A(5)     ← 3音
B:  D(3), E(5)           ← 2音 (常に)
G:  A(2), B(4), C(5)     ← 3音
D:  E(2), F(3), G(5)     ← 3音
A:  B(2), C(3), D(5)     ← 3音
6E: F(1), G(3), A(5)     ← 3音

Diminished Positions (8-Note Scales)

ディミニッシュ (W-H / H-W) は対称スケールのため 4ポジション × 2フェーズ = 8ポジション を生成。 dim7 の構成音 (Root, ♭3, ♭5, 6) は短3度間隔で等分布。

  • 各ポジションは5フレット幅、3フレット間隔
  • ポジション順: Pos 1=Root, Pos 2=♭3, Pos 3=♭5, Pos 4=6
  • 6E弦のみ4音、他弦は3音

Phrase Generator — Overview

フレーズジェネレーターは、スケルトン駆動アーキテクチャでビバップフレーズを生成します。 buildSkeleton() が表拍にCT、強拍にGT (3rd/7th) を事前配置し、コンター曲線に沿った構造的骨格を構築。 その骨格をテンプレートとセグメント関数で肉付けする2段階方式により、 §1ルール (CT表拍配置、GT強拍配置) が構造的に保証されます。 ビバップスケール、Musical Forces スコアリングとの組み合わせで、音楽理論に基づいた自然なビバップラインを生成します。 生成失敗時は旧 buildPhrase() フォールバック → 最終フォールバック (ScaleRun) の段階的退避を行います。

機能

  • ルールベース生成: 11種のテンプレートと9種のセグメント関数でフレーズを構築
  • 可変リズム: 4分音符(q)、3連符(t)、8分音符(e)、16分音符(s)の混合に対応
  • 拍数選択: 辞典モードで2/3/4拍のフレーズ長を選択可能
  • ゴールノート選択: 指板クリックでゴール音を指定し、フレーズの到達点を制御
  • ピアノロール: PhraseAnalysisPanel に時間×ピッチのSVG可視化を追加
  • セグメント境界表示: フレーズ内の各セグメント境界を分析テーブルに表示。ピアノロールにも境界の垂直破線を描画
設計哲学
ビバップ構造ルール (David Baker, Hal Galper, Anton Schwartz 等の教育的コンセンサス) に基づき、 テンプレートとセグメント関数の組み合わせで音楽的なフレーズを構築。 Musical Forces (Steve Larson) による重力・磁力・慣性のスコアリングで自然な音選択を実現。

アーキテクチャ

  • スケルトンビルダー (skeleton.ts): buildSkeleton() が表拍にCT、強拍にGT (3rd/7th) を配置するスケルトン (PhraseSkeleton) を構築。 chooseContour() でstart/goalのピッチ関係からコンターを動的に導出。 contourCurve() で各拍位置のターゲットピッチを計算し、最も近いCT/GTをアンカーとして配置。 SkeletonSlot のロール: start (開始音), downbeat-ct (表拍CT), strong-gt (強拍GT), target (ゴール音)。 ビバップパッシングトーン排除 + 演奏可能性 (ピッチ距離≤12半音, 弦距離≤3) を構築段階で検証。 §1ルール (CT表拍配置 + GT強拍配置) が構造的に保証される
  • ビバップスケール (bebopScales.ts): Dominant / Major / Dorian / Harmonic Minor の4種。モード→ビバップスケールマッピング。Diatonic 7 + Lydian Dom + Altered の9モード対応
  • セグメント関数 (bebopSegments.ts): Arpeggio, ScaleRun (表拍GT優先, ウェイポイントキュー方式), Enclosure (4タイプ+Delayed Resolution), 1-2-3-5, dim7From3rd, UpperStructure, ApproachCT (6タイプ+Musical Forces), Chromatic, OctaveDisp (Honeysuckle Rose) の9種。全て PoolNote[] → PoolNote[] で指板空間内動作。 SegmentOpts に exitAnchor (セグメント出口アンカー), interiorAnchors (中間アンカー), beatPhase (0-3の4位相) を新設。 ScaleRun: interiorAnchors → exitAnchor のウェイポイントキューを構築し、各ウェイポイントに接近時にsnap+方向更新。 スケルトンのアンカーに向かってノートを生成することで、骨格との整合性を維持
  • テンプレート (bebopTemplates.ts): 11個のフレーズ構造テンプレート (Arp↑+Scale↓, Scale↓, Honeysuckle 等)。品質フィルタ・コンター親和 (+8)・重み付きランダム選択
  • スケジューラー (bebopScheduler.ts):
    • fillSkeleton(): スケルトン駆動でセグメントを実行するメイン関数
    • distributeAnchors(): スケルトンの内部スロットを各セグメントに割当 (exit / interior)
    • planSegmentRhythms(): セグメント生成前にリズムを決定
    • エンクロージャーセグメントは16分圧縮しない (強拍アラインメント保持)
    • スケルトン整合性チェック (required=trueスロットの検証) + 安全弁チェック (音域/跳躍/方向転換) を適用
    • テンション終止はゴールノート選択段階に移動 (fillSkeleton 内では処理しない)
    • assignRhythms() は削除済み (deprecated → removed)
    • 旧 buildPhrase() はフォールバック用に残存
  • Musical Forces (§9 Steve Larson): bestCandidate() に重力 (+5/-3)・磁力 (+5)・慣性 (+3) のスコアリングを加算。±4スコアジッター (stepwise/approach選択時のみ) でピッチ列多様性を確保
  • エントリポイント (bebopGenerator.ts): generatePhraseRule() — buildNotePool → ゴール選択 → テンション終止判定 (26%, goalIsVL時は抑制) → スケルトン構築 (2回リトライ) → テンプレート選択 → fillSkeleton → 成功 or 旧buildPhraseフォールバック (3回リトライ) → 最終フォールバック (ScaleRun↓)。 SkeletonMeta に実データ (noteName, role, beatPos, ctLabel) を格納。goalReason に次コード情報を enriched

bebop-construction-rules.md 準拠改善

  • アプローチノート多様化 (§3): segApproachCT が6タイプのアプローチを WJD 統計ベースの重みで選択。 chrom-below (w=22), diatonic-above (w=25), chrom-above (w=16), diatonic-below (w=12), dbl-chrom-below (w=8), dbl-chrom-above (w=7)。§3 (Anton Schwartz): 上方アプローチはdiatonic優先、chromatic は慎重に使用。Musical Forces (重力/磁力/慣性) でCTターゲット選択を最適化。
  • エンクロージャーリライト (§4): segEnclosure を buildEncl() ヘルパー + フォールバックチェーンで再構築。 Mixed (w=55), Diatonic (w=25), Chromatic (w=20), 3-note (w=15) の重み付き選択。 synthChromNote() でプール外のクロマチックアプローチ音を合成 (fretはプール範囲にクランプ)。 Mixed エンクロージャー: プールに上行スケール音がない場合はdiatonicを合成。下方クロマチックはスケール外を検証。 beatPhase パラメータで強拍アウェア・パディング (4位相対応)。 Delayed Resolution (30%) でターゲットを裏拍にずらすバリエーションも生成。 fillSkeleton内ではエンクロージャーの16分圧縮を行わない (スケルトンの強拍アラインメントを保持)。
  • CT終止 → テンション終止 (§6): テンション終止判定をスケジューラーのポスト処理からゴールノート選択段階に移動。 goalIsVL=true (ボイスリーディング解決ゴール) 時はテンション終止を抑制し、解決の正確性を維持。 CT終止の26%を隣接非CTスケール音にスワップ (テンション終止) し、CT終止率~70%を実現 (WJD 63.7%に近接)。 fillSkeleton内でのCT終止スワップは廃止 (スケルトンのターゲットスロットが正確な音を保持)。
  • 休符/ブレス挿入 (§10): 多セグメントテンプレートのセグメント接合部で15%確率で8分/4分休符を挿入。 休符後の次セグメント開始は70%裏拍/30%表拍 (フレーズ開始と同じ確率)。全フレーズの約11%に休符が出現。 オーディオ・SVG・ピアノロール・フレーズ分析すべてで休符に対応。
  • 開始拍分布 (§6): startHint なし (単体生成) 時に 70% で裏拍 (beatOffset=0.5) 開始。 接続フレーズ (startHint あり) ではギャップなし (beatOffset=0)。 beatParity をセグメント関数に伝播し、ScaleRun/Enclosure のビバップパッシングトーン/CT 表拍判定を正確化。
  • 強拍ガイドトーン優先 (§1): 開始音は 3rd/7th に2倍の重み。スケールラン中も表拍でGT候補があれば優先選択。品質チェックで強拍 (beat 1, 3) が2つ以上ある場合にGTゼロならソフト reject。
  • 方向転換強拍チェック (§9): 方向転換の60%超が強拍 (beat 1, 3) に集中するフレーズを reject。Barry Harris / Jens Larsen ルール準拠。
  • コンター選択 (§5): スケルトン構築時にstart/goalのピッチ関係から動的に導出。下行時は descending(40)/arch(35) 優先、上行時は ascending(35)/reverse-arch(30) 優先。前フレーズと同コンター連続時は-50%減衰。旧固定重み (arch33/desc28) からピッチ関係ベースに変更。

Skeleton Builder

skeleton.ts はフレーズ生成の核となるスケルトン構築モジュールです。 セグメント実行の前に、表拍にコードトーン (CT)、強拍にガイドトーン (GT = 3rd/7th) を配置する骨格を構築します。 これにより、§1ルール (CT表拍配置 + GT強拍配置) が構造的に保証されます。

buildSkeleton()

スケルトン構築のメイン関数。以下の手順で PhraseSkeleton を生成:

  1. コンター選択 (chooseContour()): start/goal のピッチ関係から動的に導出
    • 上行 (diff > 2): ascending(35), reverse-arch(30), wave(20), arch(15)
    • 下行 (diff < -2): descending(40), arch(35), wave(15), reverse-arch(10) — §2ビバップ下行バイアス準拠
    • 同ピッチ帯: descending(30), arch(28), wave(20), reverse-arch(14), ascending(8)
    • 前フレーズと同コンター連続時は -50% 減衰ペナルティ
  2. ビートグリッド計算: beatOffset〜lastBeat の範囲で表拍 (integer beats) と強拍 (beat % 2 === 0) を列挙
  3. ハードアンカー配置: start (開始音, required=true) と target (ゴール音, required=true) を固定配置
  4. 表拍CTアンカー配置: 各表拍で contourCurve() が返すターゲットピッチに最も近いCTをアンカーとして配置。 findAnchorCT() はピッチ近接度、前アンカーからの到達可能性、弦距離をスコアリング
  5. 強拍GTバイアス: 強拍 (beat 1, 3) ではCTよりGT (3rd/7th) を優先。findGTNear() が最適なGTを探索し、 見つかれば strong-gt ロールで配置
  6. ビバップパッシングトーン排除: getBebopPassingTone() で特定されたパッシングトーンはスケルトン上に配置しない
  7. 演奏可能性検証: 連続スロット間で pitchDist > 12 または stringDist > 3 の場合、 non-required スロットを代替CTに置換またはスロット除去

コンター曲線 — contourCurve()

位置 t (0〜1) におけるターゲットピッチを返す5種の曲線関数:

fillSkeleton() — スケルトン肉付け

bebopScheduler.ts に実装。スケルトンの骨格にセグメント関数でノートを充填:

  1. distributeAnchors(): 内部スロット (start/target以外) を各セグメントのビート範囲に割当。 セグメント末尾付近のスロットは exit アンカー、それ以外は interior アンカーとして配分
  2. 各セグメントに exitAnchor / interiorAnchors を渡して実行
  3. セグメント接合部の重複除去・休符挿入 (15%)
  4. リズム決定: planSegmentRhythms() + ポスト圧縮 (chromaticのみ16分, エンクロージャーは圧縮しない)
  5. CT/GT促進パス (スケルトンが保証しきれなかったケースの軽量フォールバック)
  6. CT終止試行 (テンション終止はゴール選択段階で処理済み)
  7. ビート予算トリミング
  8. スケルトン整合性チェック: required=true のスロット (start/target) が出力に含まれるか検証
設計判断: エンクロージャーの16分圧縮を廃止
旧 buildPhrase() ではエンクロージャーセグメントの35%を16分音符に圧縮していたが、 fillSkeleton() ではこれを行わない。理由: スケルトンが強拍にGTアンカーを配置しており、 エンクロージャーの圧縮は強拍アラインメントを崩す可能性があるため。

Instance Scope Restriction

音域をコンパクトに保つため、第1インスタンスの範囲に制限します。

  • activePool: instFretMin - 1 〜 instFretMax + 1 のノートのみ
  • activeCtPool: activePool のうちコードトーンのみ
  • プール不足時 (≤6音 or CT<2) は全プールにフォールバック

Goal Note Selection

フレーズの最終音（beat 8）を最初に決定します。これが全体の方向性を決めます。

設計根拠
  • 70% の理由: パーカーの実際のソロでは ii-V-I フレーズ境界で 60-80% 程度の 7th→3rd 半音解決が観察される。100% だとフレーズが画一化するため 30% の変化余地を残す。
  • 半音（1 semitone）のみ: diff === 1 || diff === 11 で厳密に半音差のみチェック。全音は含まない。ii→V、V→I のような典型進行でのみ発動し、半音解決がない進行ではフォールバックが機能する。
  • 辞典モードのゴール音: 次コードの 3rd そのものではなく、STRONG_RESOLUTION_DEGREE_IDX で推定した解決先の度数に最も近い現コードの CT（特に 3rd/7th）をゴールにする。ビバップでは現コード内で解決方向を示唆するのが定石。

chooseGoalNote() の決定ロジック

練習モード（nextChordContext + nextChordPool あり）
  1. 現コードの7thが次コードの3rdに半音で解決する場合 → 70%の確率で現コードの7thをゴールに
  2. 次コードの3rdと1半音差のCTを優先検索
  3. フォールバック: 次コードの3rdと同名のCTを選択
  4. resolvedNextStart: ゴール音に absolutePitch が最も近い次コードプール内の音を確定。次コードのポジション fretMin〜fretMax 内かどうかで inPosition を設定
コード間ボイスリーディング接続 (resolvedStart)

演奏者の思考を再現する VL 意図の伝達メカニズム:

  1. コードA: chooseGoalNote() が resolvedNextStart を確定（次コードの解決先 PoolNote + inPosition フラグ）
  2. コードA: フレーズ生成、ゴール到達時に resolvedGoalForNext を返す
  3. コードB: config.resolvedStart として受け取り、startHint より優先

inPosition=true: 解決音をそのまま次フレーズの開始音に使用（シームレス接続）

inPosition=false: 解決音を1音(8分音符)で鳴らし、8分休符を挿入してからポジション内で本体開始（演奏者の「解決→ブレス→新ポジション」の動き）

辞典モード

STRONG_RESOLUTION_DEGREE_IDX で「このモードが解決する先のコードの3rd」を推定。実際のゴール音は現コードのCTから選択:

ModeResolutionTarget
IonianI → IVIVの3rd (scale deg 6)
Dorianii → VVの3rd (scale deg 7)
MixolydianV → IIの3rd (scale deg 3)
Aeolianvi → iiiiの3rd (scale deg 4)

Audio Engine

すべて Web Audio API ベース。外部ライブラリ依存なし。

Design Decisions

  • chords: ChordSlot[] フラット配列 = ロジックのsingle source of truth
  • chartLayout?: ChartLayout = 表示用メタデータ（chords[] へのインデックス参照）
  • ユーザー作成進行（layoutなし）→ deriveChartLayout() で1コード/小節に自動生成
  • beatWidths: フレックス比率 — ChordChart で flex: beats, BPM 再生で (bw/bwSum)*4

chartLayout Preservation on Edit

以前はコードの追加・削除・編集時に chartLayout を破棄（undefined にセット）していたため、 インポートした曲の小節構造が失われていた。現在は編集操作ごとに chartLayout を差分更新して構造を保持する。

  • 小節追加: 「+ 小節」ボタンで空小節を追加し、即座にビートグリッドの挿入モードに入る。insertEmptyMeasure(layout, afterChordIdx?)
  • コード挿入: ビートグリッドの「+」をクリックして拍位置を選択 → コード名を入力 → Enter で挿入。insertChordAtBeat() / fillEmptyMeasure()
  • コード削除: removeChordFromLayout(layout, deletedIdx) — 該当コードを小節から除去し、後続の全 chordIndices をデクリメント。小節が空になっても小節は残る（空小節として表示）
  • 空小節削除: 空小節の右端に表示される縦長の × ボタンで明示的に削除
  • コード更新: インデックス不変のため chartLayout は変更不要
  • セクション分割: splitSection(layout, sectionIdx, measureIdx) — 指定小節位置でセクションを2つに分割。新セクションには自動ラベル (A-Z) を付与
  • セクション結合: mergeSections(layout, sectionIdx) — 隣接する2セクションを結合。最初のセクションのラベル・リピート設定を保持
  • リピート設定: section.repeats を undefined ↔ 1 でトグル。BPM再生時にセクション全体が2回演奏される

Beat-Position Chord Insertion

編集モードでは ChordChart が4列のビートグリッドを表示し、小節内の特定拍位置にコードを挿入できる。 空き拍に + マーカーが表示され、クリックでその拍位置にコードを挿入する。

  • getMeasureBeatInfo(layout, chordIdx) — 指定コードが属する小節の占有拍数と総拍数を返す
  • computeInsertFlatIndex(layout, referenceIdx, beat) — 小節内の拍位置からフラット配列上のスプライス位置を算出。既に占有されている拍位置には null を返す
  • insertChordAtBeat(layout, referenceIdx, beat, newFlatIdx) — 指定拍位置に新コードを挿入し、beatWidths を比率正規化（合計4拍/小節）して ChartLayout を返す
ビートグリッド表示

編集モード時、ChordChart の各小節は4列グリッドに分割される。 コードが占有する拍はコード名を表示し、空き拍には + ボタンを配置。 beatWidths は常に合計4拍に正規化される。

Section Structure — Symbol Buttons

編集モードでコードの1拍目をクリックすると、ツールバーに記号ボタン行が表示される。 ボタンを押すとその小節にマーカーをスタンプし、必要に応じてセクション分割/結合が自動実行される。 ChordChart 自体は表示専用（リピートドット・ボルタ括弧は自動描画）。

セクション名入力 + 記号ボタン4種:

  • Sec: (セクション名): セクション先頭の小節ではラベルを直接編集。それ以外の小節でラベルを入力するとそこでセクションを自動分割し新セクション名を設定。先頭小節のラベルを空にすると前セクションに結合。splitSection() / mergeSections() / renameSection()
  • |: (リピート開始): 選択小節がセクション先頭ならリピートをトグル。先頭でなければセクションを分割し新セクションにリピートを設定。splitSection()
  • :| (リピート終了): 選択小節がセクション末尾ならリピートをトグル。末尾でなければセクションを分割し前半にリピートを設定。splitSection()
  • 1. (1ボルタ): リピートONのセクション内で、選択小節からボルタ1括弧を設定（残り小節は全て1括弧に入る）。2括弧は別途「2.」で設定。既に1ボルタ内の小節をクリックすると解除。splitEndings() / removeEndings()
  • 2. (2ボルタ): エンディング内の小節をクリックし、その小節から2番カッコを開始。1括弧のみの状態からでも分割可能。adjustEndingSplit()

ボタンの点灯状態: 選択中の小節に既にそのマーカーがある場合は色付き（|:/:| = 黄, 1. = 青, 2. = 緑）。 エンディング内の小節では |:/:| は無効。リピートなしでは 1. は無効。エンディングなしでは 2. は無効。

  • コンタープレビュー: LickNote[] の pitch を SVG ポリラインで描画。フレーズカラー #FF6B9D、選択時は不透明、未選択時は半透明
  • 折りたたみ時: 選択中リックのID + ▶/■ + ✕ ボタンをインライン表示。未選択時は「クリックで展開」ヒント
  • リック選択時に指板にフレーズ曲線を即時表示 + 自動再生
  • 分析パネル (PhraseAnalysisPanel) はそのまま動作（度数/インターバル/機能ラベル）

辞典モードではフレーズ生成UIを完全に非表示（OptionBar の「フレーズ」チェックボックス削除、PhraseControls 無効）。 辞典モードはモード/ポジション閲覧専用として機能。指板クリック音も無効化され、GlobalAudioControls（再生バー）も非表示。 ガイドトーンは辞典モードで常時表示され、現モードの 3rd/7th を指板上に表示（次コード情報はなし）。
